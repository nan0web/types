# Internationalization (i18n)

The `@nan0web/types` package provides a lightweight i18n engine designed for both simplicity and architectural robustness in high-performance JS applications.

## `createT(vocabulary, locale)`

Creates a translation function that supports variable substitution, recursive pluralization, and numeric shorthands.

```js
import { createT } from "@nan0web/types"
const t = createT({
    'Hello {name}': '{name}, hello!',
    apples_one: '{count} apple',
    'I have {apples}': 'I have {apples}',
}, 'en-US')

console.info(t('Hello {name}', { name: 'World' })) // "World, hello!"
console.info(t('I have {apples}', { apples: 5 })) // "I have 5 apples"
```

### `TFunction` (Contract)
A function type definition for translations:
`(key: string, vars?: Record<string, any>) => string`

## 🏗 Symmetrical i18n Validation

This package establishes a pattern for **Symmetrical i18n Validation**. 

### The Problem
Traditional validation returns static strings, which are hard to translate in the UI layer (CLI, Web, Mobile).

### The Solution: "Domain-First Errors"
Instead of returning static strings from validation rules, your domain models return **translatable structures** (e.g., `["Invalid DSN: {dsn}", { dsn: value }]`).

These errors are:
1.  **Defined** in domain models using simple arrays/objects.
2.  **Caught** as `ModelError` by the application.
3.  **Translated** at the UI layer using a local `TFunction`.

### `ModelError`
A structured error class for validation failures. It holds a map of fields that failed validation and their respective error messages or translation configs, preserving the context ($ prefix) for pluralization and substitution.

```js
import { resolveValidation, ModelError } from "@nan0web/types"
class User {
    static name = { validate: (v) => v.length > 2 || 'Name too short' }
}

try {
    resolveValidation(User, { name: 'Bo' })
} catch (e) {
    if (e instanceof ModelError) {
        console.info(e.fields) // { name: "Name too short" }
    }
}
```
